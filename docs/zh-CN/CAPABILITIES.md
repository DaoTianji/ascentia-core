# 能力与设计理念

本文说明 **ascentia-core 能做什么**、**为何可称为 LLM/智能体 harness（运行架）工程**，以及 **记忆、思考、反思** 等叙事在代码中的对应关系。

---

## Harness 工程：这个仓库在解决什么问题

在生产环境里，**模型权重 + 一次 HTTP 调用**并不等于产品。真正上线的是 **harness（运行架）**：围绕模型的一整套 **约束、编排与观测**——会话、工具、策略、记忆、重试、面向客户端的流式输出、用量归因等。

**ascentia-core 的定位就是「智能体运行时 harness」**：

| 关注点 | 在本项目中的落点 |
|--------|------------------|
| 谁在说话、是哪个助手实例？ | `TenantScope`、WebSocket 握手参数 / 鉴权 |
| 本轮模型看到什么上下文？ | 提示词拼装、上下文压缩、旁路记忆召回、Todo 焦点 |
| 如何安全地多轮用工具？ | ReAct 循环、`MAX_TURNS`、连续工具失败熔断 |
| 多轮对话状态放哪？ | STM（Redis 或进程内）、条数与 TTL 上限 |
| 哪些信息要跨会话保留？ | 可选 PostgreSQL LTM、读写工具、回合后抽取 |
| 如何控制成本与延迟？ | 上下文 token 预算、工具结果截断、HTTP 超时 |
| 客户端如何消费输出？ | WebSocket JSON 帧、流式 assistant 增量 |

**大模型**以 **OpenAI 兼容 Chat Completions** 形式接入；**harness** 负责 **调用前后与多轮之间的控制面**。

---

## 能做什么（功能地图）

### 1. 对话式智能体循环（ReAct）

- 支持 **工具调用** 的多轮循环，并在每一轮模型调用中 **流式**输出助手文本。
- 可配置 **最大工具轮次**（`MAX_TURNS`）与 **连续工具失败熔断**（`MAX_TOOL_FAILURE_STREAK`），防止死循环烧券。
- 内置 **规划类工具**（`update_todo` / `mark_task_done`），把任务状态外置到 `planner` 存储，减轻模型「记流水账」负担。

### 2. 提示词与上下文工程

- **模块化系统提示**：核心规则、人设 (Persona)、时间上下文、可选 **记忆召回段落**、可选 **Todo 焦点段落**。
- **上下文压缩**（`CONTEXT_TOKEN_BUDGET` 等）：在预算内保留尾部消息，控制超长对话成本。
- **工具结果裁剪**（`TOOL_RESULT_MAX_RUNES`）：避免巨型工具输出撑爆上下文。

### 3. 记忆：短期 + 长期 + 反思链路

| 层次 | 机制 | 适用场景 |
|------|------|----------|
| **STM（短期）** | Redis 列表（或无 Redis 时进程内 map），按租户+会话隔离 | 多轮对话连续推理，无需客户端每次重传全量历史 |
| **LTM（长期，可选）** | PostgreSQL，按 session + `user_id` + `agent_id` 隔离 | 跨会话偏好、事实、用户档案片段 |
| **旁路召回（Side Query）** | 每轮对话前按查询从 LTM 取片段写入提示词 | 无需客户端自建 RAG 管道即可「带着记忆说话」 |
| **回合后抽取（Post-turn）** | 成功回合后异步小模型调用（`LLMExtractor`）写入 LTM | 从闲聊中沉淀稳定事实，偏「潜意识写入」 |
| **Dream 整理** | 定时 LLM 整理（`DreamConsolidator`）合并/去噪 | 长期记忆库随时间保持可读、减少矛盾与重复 |

上述路径均带 **租户隔离**，与 `user_id` / `agent_id` / 会话命名空间一致。

### 4. 网关与运维向能力

- **WebSocket + JSON** 协议，便于 Web / 移动端 / 自有 BFF 对接。
- **鉴权**：`none`（可信网络/开发）、静态 Bearer、HS256 JWT（顶层数值 `ID`）。
- **Origin 策略**、可选 **按 IP 的每分钟消息限流**。
- **用量归因**上下文，便于 token 账本与审计字段。

### 5. 「思考」流（引擎能力）

`pkg/agent_core/thinking` 支持在流式输出中将 **`<thinking>…</thinking>`** 与对用户可见正文拆分，便于独立展示或记日志。默认 HTTP 适配到 WS 的链路**未必**已端到端暴露给前端；该能力可通过自定义 `ModelClient` / `StreamSink` 接入。

### 6. 可选演示向集成

- **NATS** 示例工具 SpawnPet（可用 `DISABLE_SPAWN_PET=1` 关闭）。
- **Redis / NATS** 可选 **配置总线**，对白名单环境变量做热更新（进阶运维）。

## 优势总结（适合谁用）

- **分层清晰**：`pkg/agent_core` 不依赖 HTTP/DB；适配器集中在 `internal/`，便于阅读和二次封装。
- **多租户内建**：`TenantScope` 校验并贯穿记忆与工具上下文。
- **运维护栏齐全**：轮次上限、熔断、压缩、限流——面向 **服务** 而非笔记本脚本。
- **记忆体系完整**：STM + LTM + 召回 + 回合后抽取 + 定时整理，不是「只把历史拼成字符串」。
- **可部署**：单 Go 二进制、环境变量驱动、易于接 CI。

---

## 非目标（边界说明）

- 不是完整 SaaS（注册登录、计费、可视化编排等需自建）。
- 不是开箱即用的 **评测 harness**（黄金集、自动打分需自行扩展）。
- 不是 LangChain 类「编排 DSL」的替代品；侧重 **可部署运行时 + 网关**。
- JWT 载荷形态有约定（`ID`）；异构 IdP 建议在网关映射或扩展解析。
